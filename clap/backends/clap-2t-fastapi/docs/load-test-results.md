## 3. 종합 분석

### 성능 비교

| 항목 | 텍스트 | 오디오 |
|------|--------|--------|
| 피크 VU | 30 | 10 |
| 처리량 | 12.31 req/s | 3.94 req/s |
| p(95) | 2.33s | 2.36s |
| 에러율 | 0% | 0% |
| 네트워크 (송신) | 192 kB | 523 MB |

### 주요 관찰

1. **안정성**: 두 엔드포인트 모두 에러율 0%, Checks 100% 통과로 완전한 안정성 확인.

2. **아키텍처 특성**: gateway(librosa 전처리) → HTTP → model(CLAP 추론) 구조. gateway는 동시성 제한 없이 요청을 받아 model 서비스로 전달하며, `semaphore=2` 제약이 model 서비스에 위치하여 gateway의 수평 확장이 자유롭다.

3. **텍스트 레이턴시 특성**: 피크 30 VU에서 p(95) 2.33s. semaphore가 model 서비스에서 관리되므로 gateway는 큐잉 오버헤드 없이 즉시 model에 포워딩하며, HTTP 통신(gRPC 아님)임에도 clap-triton 대비 낮은 레이턴시를 달성.

4. **오디오 레이턴시 특성**: p(95) 2.36s, 최대 7.24s로 텍스트보다 최대값이 높음. WAV 파일 전송(523 MB 총 송신)이 HTTP 페이로드 처리 부담을 가중시키지만, gateway에서 librosa 전처리 후 전처리된 데이터만 model로 전달하는 구조로 전체 레이턴시를 효율적으로 관리.

5. **clap-triton 대비 우위**: Triton gRPC 오버헤드 없이 직접 CLAP 모듈을 model 서비스에서 호출하므로 추론 경로가 단순. 2-tier 구조임에도 HTTP 포워딩 비용이 gRPC + Triton 오버헤드보다 낮아 처리량이 높음.

6. **네트워크 비대칭**: 오디오는 송신 523 MB (WAV 파일 업로드) vs 수신 4.3 MB (임베딩 응답). 텍스트는 역방향으로 송신 192 kB (텍스트) vs 수신 12 MB (임베딩 응답). 오디오 테스트 시 업로드 대역폭이 병목이 될 수 있음.
