### VU 구간별 추이

```
VUs  10  → 30              → 60 (피크)        → 30        → 0
     ↑warm-up  ↑압박 시작        ↑포화 구간          ↑회복

```

### 분석

- **60 VUs 연속 단건 요청에도 에러율 0%** — 서버가 큐를 소진하지 않고 모든 요청을 처리함.
- **처리량 포화점**: 약 9~10 req/s. 60 VUs에서도 throughput이 선형 증가하지 않고 수렴 → CPU가 병목.
- **지연 vs. 에러의 트레이드오프**: FastAPI의 sync worker가 큐를 직렬 처리하므로 VU가 늘수록 대기 누적 → p95 6.78s. 하지만 에러는 0건 — graceful degradation 상태.
- **배치 제거 효과**: 이전 배치(2~5개) 테스트 대비 단건이 per-request 지연은 낮지만 총 VU 압박이 훨씬 커서 전체 p95는 높음.

## 4. 서버 한계 분석

### 텍스트: CPU-bound 포화, 메모리 안정

```
안전 운영 구간:  VUs ≤ 10  →  p95 < 3.5s, 에러 0%
경계 구간:       VUs 10~30  →  p95 3~5s, 에러 0%
포화 구간:       VUs 30~60  →  p95 5~7s, 에러 0% (큐잉, graceful)
한계 도달 기준:  에러 없이 지연만 증가 → 스케일아웃 필요 시점
```

### 오디오: 메모리-bound 충돌

```
안전 운영 구간:  VUs ≤ 10  →  p95 < 1s, 에러 0%
위험 구간:       VUs 10~15  →  메모리 압박 증가 (미측정)
한계 도달:       VUs ≈ 20   →  OOMKill (8,000Mi 초과)
```

## 5. 개선 방향

| 문제 | 원인 | 해결책 |
|------|------|--------|
| 텍스트 p95 증가 (60 VUs) | CPU 직렬 추론 + 큐잉 | replicas 증설 or `run_in_executor` 비동기화 |
| 오디오 OOMKill (20 VUs) | 동시 추론 시 activation 메모리 누적 | ✅ **해결 완료** — `torch.no_grad()` + `threading.Semaphore(2)` + `del raw` 적용 |
| 오디오 메모리 비효율 | `librosa.load()` 32bit float 버퍼 | `dtype='float16'` 또는 서버사이드 오디오 청크 캐싱 |
| 단일 Pod 의존 | OOM 시 재시작 ~1분 다운타임 | `minAvailable: 1` PodDisruptionBudget + replicas 2 |

## 6. OOMKilled 해결 및 재테스트

| # | 변경 | 내용 |
