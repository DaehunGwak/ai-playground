## 3. sleep 유무 비교

### 텍스트 (`/embed/text`, 10 VUs)

| 지표 | sleep(0.5s) | sleep 없음 | 변화 |
|------|------------|-----------|------|
| 총 요청 수 | 293 | 304 | +3.8% |
| Throughput | 3.64 req/s | 3.80 req/s | +4.4% |
| avg | 1.93s | 2.34s | **+21%** |
| median | 2.02s | 2.40s | +19% |
| p(90) | 2.89s | 3.42s | +18% |
| p(95) | 3.02s | 3.57s | **+18%** |
| max | 3.45s | 3.87s | +12% |
| 에러율 | 0% | 0% | — |
| 임계값 통과 | ❌ | ❌ | — |

> **관찰**: sleep 제거 시 throughput 증가는 미미(+4.4%)한 반면 지연은 +21% 악화.
> 원인: 이미 응답 시간이 ~2s로 긴 상황에서 sleep(0.5s)는 전체 iteration의 ~21%에 불과해 제거 효과가 제한적. 반면 VU들이 응답 즉시 다음 요청을 보내 서버 큐가 더 빠르게 적체됨.

### 오디오 (`/embed/audio`, 3 VUs)

| 지표 | sleep(1s) | sleep 없음 | 변화 |
|------|----------|-----------|------|
| 총 요청 수 | 151 | 357 | **+136%** |
| Throughput | 1.87 req/s | 4.45 req/s | **+138%** |
| avg | 433ms | 603ms | +39% |
| median | 435ms | 617ms | +42% |
| p(90) | 570ms | 818ms | +43% |
| p(95) | 596ms | 843ms | +41% |
| max | 2.02s | 907ms | -55% |
| 에러율 | 0% | 0% | — |
| 임계값 통과 | ✅ | ✅ | — |

> **관찰**: sleep(1s)이 응답시간(433ms)의 2.3배였으므로, 제거 시 throughput이 2.4배 급증.
> 지연은 +41% 증가했으나(596ms → 843ms) 임계값(15s) 대비 여전히 매우 여유. 에러 0건으로 3 VUs 연속 요청은 서버가 충분히 소화.
> max가 2.02s → 907ms로 감소한 것은 이전 테스트의 outlier가 사라진 것으로, 실제 안정성이 오히려 개선된 패턴.

## 4. 종합 분석

### 병목 위치 — 텍스트

```
10 VUs × 연속 요청 → FastAPI worker 경합 → 큐 대기 증가
                                         ↓
                          CLAP 모델 추론 (CPU 단일 코어 직렬화)
                                         ↓
                          p95 3.57s (임계값 초과)
```

텍스트 임베딩은 **모델 추론이 CPU-bound**이므로, 단일 Pod에서 10 VUs 동시 요청 시 큐잉이 발생. sleep 제거로 요청 간격이 0이 되면 CPU가 포화 상태를 유지해 지연이 선형 이상으로 증가.

### 병목 위치 — 오디오

```
3 VUs × 연속 요청 → 처리량 4.45 req/s
                  ↓
    avg 603ms / p95 843ms / 에러 0건 → 안정 구간
```

오디오는 3 VUs 연속 요청에서도 p95 < 1s로 안정적. **VU 수가 낮아 CPU 경합 임계 이하**를 유지하며, sleep 제거가 오히려 잠재 처리량을 끌어냈음.

## 5. 개선 방향

| 개선안 | 예상 효과 | 대상 |
|--------|---------|------|
| **replicas 2~3으로 증설** | 텍스트 p95 ~1.5s 예상 (CPU 부하 분산) | 텍스트 ❌ 해소 |
| **FastAPI `asyncio.run_in_executor`로 추론 오프로드** | worker 블로킹 해소, concurrency 향상 | 텍스트 |
| **오디오 VU 5~10으로 증설 테스트** | 현재 3 VUs는 여유, 실제 한계점 미탐지 | 오디오 여유 확인 |
| **배치 크기 상한 제한 (max 3)** | 텍스트 worst-case 지연 감소 | 텍스트 p95 |
| **WAV → MP3 업로드** | 송신 473MB → ~50MB로 감소 | 오디오 네트워크 |
